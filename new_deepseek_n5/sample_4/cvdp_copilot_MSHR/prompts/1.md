## Functional Description:

- The design includes a register for each MSHR entry for meta data with the following fields:

| Field           | Description                                                                      |
|-----------------|----------------------------------------------------------------------------------|
| valid           | Indicates if the entry is valid (active).                                        |
| cache line addr | Stores the cache line address associated with the request.                       |
| write           | Specifies the request type: 1 for write (RW) and 0 for read (RD).                |
| next            | Indicates if there is a valid next entry linked to this one.                     |
| next index      | Points to the index of the next entry in the MSHR linked to the same cache line. |

- The design utilizes a single-port RAM to store MSHR entry data, featuring a write latency of 1 cycle and a read latency of 1 cycle.
- This design should integrate seamlessly into a non-blocking cache system with 32-bit address system, where **tag access** and **MSHR access** occur in parallel. This parallel operation minimizes the critical path delay associated with tag lookup and MSHR operations by allowing the system to simultaneously determine whether the requested data is already in the cache (tag lookup) and whether there is an existing outstanding request for the same cache line in the MSHR.

### Within cache pipeline, on a core request to memory:
- In the first cycle, the MSHR must allocate **the first available** entry for the incoming core request, regardless of whether it is a hit or miss.
    - If the requested cache line address is marked as pending it means it's an outstanding miss(a cache line that is already being handled, no need to wait for tag access) thus the module sets the next pointer of the previous index to the newly allocated slot. , employing a linked list structure. 
    - The module uses leading zero counter module provided to get:
        - The index of the first available slot to allocate.
        - The index of the previous entry requesting the same cache line  
    - Example of an MSHR entries linkage where requests to 0x44 address are organized in ordered linked list
```mermaid

    A1[index=1, addr=0x44, next_idx= 3 ] --> B1[index=3, addr=0x44, next_idx= 4 ]
    B1 --> B2[index=4, addr=0x44, next_idx= x ]
```
- In a subsequent cycle, the cache tag access determines if the request is a hit or miss. 
    - If the request is a **cache hit**, the allocated MSHR entry is marked as invalid, releasing the slot for future use.


### Latency:
- Allocation Requests: The module introduces a latency of 1 clock cycle for each allocation request.
- Finalize Requests: The module introduces a latency of 1 clock cycle for each finalize release request.
