The testbench instantiates two versions of the DUT, one configured for binary encoding and one for one-hot encoding, ensuring that both output formats are verified under the same conditions. It applies a series of test vectors that cover a range of input scenarios, including edge cases such as no set bits, a single set bit, multiple set bits, and extreme cases like all bits set or specific bit positions. The apply_test_vector task drives the inputs, toggles the In_Valid signal, and then waits for both DUT outputs to become valid, effectively accounting for pipeline delays inherent in the design. For binary encoding, the testbench computes the expected first set bit index using a helper function and compares it against the DUT's Out_FirstBit after proper zero-extension, while for one-hot encoding, it generates the expected one-hot vector and verifies it against the DUT output. The reset phase and clock generation ensure that the DUT is properly synchronized and that any transient effects from initialization or pipeline registers are cleared before applying the test vectors. Overall, this systematic approach confirms that the signals Out_FirstBit, Out_Found, and Out_Valid accurately reflect the input conditions in both encoding formats, thereby verifying the module's correct functionality under various scenarios.
